Fraxel games are standard ES module applications. They bundle the same way as any other TypeScript project. This page covers production builds with Vite, asset path conventions, and browser compatibility.

## Bundler setup — Vite

Vite is the recommended bundler for Fraxel games. It supports TypeScript and JSX out of the box, serves assets from the `public/` directory in development, and produces an optimized bundle for production.

### With @fraxel/vite-plugin (recommended)

The `@fraxel/vite-plugin` plugin auto-configures the JSX transform and enables explicit asset imports (`import tex from './sprite.png?texture'`). Projects created with `create-fraxel` include this plugin by default.

<Steps>
  <Step title="Install the plugin">
    <CodeGroup>

    ```bash npm
    npm install --save-dev @fraxel/vite-plugin
    ```

    ```bash yarn
    yarn add --dev @fraxel/vite-plugin
    ```

    ```bash pnpm
    pnpm add --save-dev @fraxel/vite-plugin
    ```

    </CodeGroup>

  </Step>

  <Step title="Create vite.config.ts">

    ```ts vite.config.ts
    import { defineConfig } from 'vite'
    import { fraxel } from '@fraxel/vite-plugin'

    export default defineConfig({
      plugins: [fraxel()],
    })
    ```

    The plugin sets `oxc.jsx.importSource` to `'fraxel'` automatically. You do not need to configure JSX options manually.

  </Step>

  <Step title="Add client types to tsconfig.json">

    ```json tsconfig.json
    {
      "compilerOptions": {
        "types": ["@fraxel/vite-plugin/client"]
      }
    }
    ```

    This enables TypeScript to recognize asset imports like `import tex from './sprite.png?texture'`.

  </Step>
</Steps>

See the [Vite Plugin Reference](/api/engine/vite-plugin) for all options and details.

### Without the plugin (manual config)

If you prefer not to use the plugin, configure the JSX transform manually:

<Steps>
  <Step title="Install Vite">
    <CodeGroup>

    ```bash npm
    npm install --save-dev vite
    ```

    ```bash yarn
    yarn add --dev vite
    ```

    ```bash pnpm
    pnpm add --save-dev vite
    ```

    </CodeGroup>

  </Step>

  <Step title="Create vite.config.ts">

    ```ts vite.config.ts
    import { defineConfig } from 'vite'

    export default defineConfig({
      // No React plugin needed — Fraxel has its own JSX runtime
      esbuild: {
        jsxImportSource: 'fraxel',
        jsx: 'automatic',
      },
      build: {
        target: 'es2022',
      },
    })
    ```

    <Note>
      Vite reads `jsxImportSource` from `tsconfig.json` for type checking, but you must also set it in `vite.config.ts` for the esbuild transform used during the dev server and production build.
    </Note>

  </Step>

  <Step title="Add scripts to package.json">

    ```json package.json
    {
      "scripts": {
        "dev":   "vite",
        "build": "tsc && vite build",
        "preview": "vite preview"
      }
    }
    ```

  </Step>

  <Step title="Build for production">

    ```bash
    npm run build
    ```

    Vite outputs your game to `dist/`. The `dist/` directory contains the HTML entry point, the bundled JavaScript, and all assets copied from `public/`.

  </Step>
</Steps>

## Asset paths

In projects created with `create-fraxel`, static assets are imported with `?texture` and `?sound`. Vite rewrites those imports for production and the Fraxel plugin decodes them into asset IDs.

Place game assets in `src/assets/` when they are known at build time:

```ts src/assets.ts
export { default as PLAYER_TEX } from './assets/sprites/player.png?texture'
export { default as JUMP_SFX } from './assets/sounds/jump.mp3?sound'
```

Set Vite's `base` option when deploying to a subdirectory:

```ts vite.config.ts
export default defineConfig({
  base: '/my-game/', // for deployment at example.com/my-game/
  // ...
})
```

<Note>
  Use runtime `loadTexture('/assets/...')` or `loadSound('/assets/...')` for
  files whose URLs are not known at build time. Put those files in `public/` so
  they are available by URL after deployment.
</Note>

## HTML entry point for production

Your `index.html` does not change between development and production. Vite handles hashing, minification, and relative path rewriting automatically.

```html index.html
<!doctype html>
<html lang="en">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <title>My Game</title>
    <style>
      * {
        box-sizing: border-box;
        margin: 0;
        padding: 0;
      }
      body {
        background: #000;
        display: flex;
        align-items: center;
        justify-content: center;
        height: 100vh;
        overflow: hidden;
      }
      canvas {
        /* Preserve pixel art sharpness when scaled up */
        image-rendering: pixelated;
        image-rendering: crisp-edges;
      }
    </style>
  </head>
  <body>
    <div id="app"></div>
    <script type="module" src="/src/main.tsx"></script>
  </body>
</html>
```

## Browser compatibility

Fraxel requires the following browser features:

| Feature        | Minimum version                     | Notes                                     |
| -------------- | ----------------------------------- | ----------------------------------------- |
| Canvas 2D API  | All modern browsers                 | `CanvasRenderingContext2D`                |
| ES2022 modules | Chrome 92+, Firefox 90+, Safari 15+ | `import` / `export`, top-level `await`    |
| Web Audio API  | Chrome 66+, Firefox 60+, Safari 14+ | Required for `loadSound` and `<audio>`    |
| Pointer Events | All modern browsers                 | `pointerdown`, `pointermove`, `pointerup` |

Fraxel does **not** support Internet Explorer.

<Tip>
  If you need to support older browsers, set `build.target` to `'es2020'` in
  your Vite config and use a polyfill for `structuredClone` if you use it
  anywhere. Top-level `await` requires `type="module"` on the script tag, which
  is already ES2022+.
</Tip>

## Preloading assets before `game.play()`

Assets loaded with `loadTexture` and `loadSound` are decoded asynchronously. If `game.play()` is called before all textures are ready, the first few frames may show missing sprites.

Because Fraxel uses top-level `await`, the recommended pattern — loading all assets before calling `createGame` — is simply how the code reads naturally:

```tsx src/scenes/main.tsx
import { loadBatchAsset } from 'fraxel'

const [PLAYER, BG] = await loadBatchAsset('texture', [
  '/assets/sprites/player.png',
  '/assets/sprites/bg.png',
])
const [MUSIC] = await loadBatchAsset('sound', ['/assets/sounds/music.mp3'])

void PLAYER
void BG
void MUSIC

export default function MainScene() {
  return <transform />
}
```

```tsx src/main.tsx
import { createGame, GameRoot, SceneRoot } from 'fraxel'

const game = createGame(
  <GameRoot width={320} height={180} defaultScene='main'>
    <SceneRoot name='main' component={() => import('./scenes/main.js')} />
  </GameRoot>,
  document.querySelector('#app')!,
)

game.play()
```

For games with large asset sets, display a loading screen by rendering a simple HTML overlay while the `loadBatchAsset` promise resolves.

## Device pixel ratio (DPR)

On high-DPI displays, the browser's device pixel ratio (`window.devicePixelRatio`) is greater than 1. The canvas logical size (your `width` / `height` in `<GameRoot>`) is always in game pixels — Fraxel handles DPR scaling internally so sprites are not blurry on Retina screens.

You do not need to multiply your canvas dimensions by `window.devicePixelRatio`. Set the game resolution you want and let Fraxel and the browser handle physical pixel scaling.

## Scaling the canvas with CSS

The canvas Fraxel creates fills the `#app` container. To make the game fill the browser window while maintaining aspect ratio, use CSS:

```css
#app {
  width: 100%;
  height: 100%;
  display: flex;
  align-items: center;
  justify-content: center;
}

canvas {
  /* Scale to fit while preserving aspect ratio */
  max-width: 100%;
  max-height: 100%;
  image-rendering: pixelated;
}
```

`Input.pointerPosition` is automatically adjusted to game coordinates regardless of CSS scaling, so click targets remain accurate at any display size.

## Deploying to static hosting

Fraxel games are static files — HTML, JavaScript, and assets. Deploy the `dist/` folder to any static host:

- **GitHub Pages** — push `dist/` to the `gh-pages` branch or configure Pages to serve from `dist/`.
- **Netlify / Vercel** — point the build output to `dist/` and set the build command to `npm run build`.
- **itch.io** — zip the `dist/` folder contents (not the folder itself) and upload as an HTML game.

<Tip>
  For itch.io, make sure the HTML file at the root of your zip is named
  `index.html` and that asset paths use relative URLs — set `base: './'` in
  `vite.config.ts` when targeting itch.io.
</Tip>
