Fraxel imposes no mandatory project structure — it is a library, not a framework. Projects created with `create-fraxel` follow the layout described below. If you set up your project manually, this structure is a recommended starting point that scales well as your game grows.

## Naming convention

Use **kebab-case** for all file and folder names: `player.tsx`, `game-scene.tsx`, `main-action.ts`. This avoids case-sensitivity issues across operating systems and keeps the project consistent.

## Recommended layout

The recommended structure assumes `create-fraxel` and `@fraxel/vite-plugin`. Static assets live in `src/` and are imported with `?texture` or `?sound`.

### Assets in `src/`

When using `@fraxel/vite-plugin`, place raw assets inside `src/` so you can import them directly:

```
my-game/
├── src/
│   ├── main.tsx              # Entry point
│   ├── style.css             # Canvas and body styles
│   ├── actions.ts            # Input.createAction — all input bindings
│   ├── constants.ts          # Game-wide constants (GRAVITY, PLAYER_SPEED, etc.)
│   ├── assets/
│   │   ├── sprites/
│   │   │   ├── player.png
│   │   │   └── enemy.png
│   │   └── sounds/
│   │       ├── shoot.mp3
│   │       └── music.mp3
│   ├── scenes/
│   │   ├── menu.tsx
│   │   └── game.tsx
│   ├── components/
│   │   ├── player.tsx
│   │   ├── enemy.tsx
│   │   └── hud.tsx
│   └── lib/                  # Utilities and helpers
├── index.html
├── tsconfig.json
└── vite.config.js
```

### Runtime URL alternative

Use `public/` for assets whose URLs are built at runtime, or for a custom integration without `@fraxel/vite-plugin`. Those assets are loaded explicitly via `loadTexture` / `loadSound`:

```
my-game/
├── public/
│   └── assets/
│       ├── sprites/
│       │   ├── player.png
│       │   └── enemy.png
│       └── sounds/
│           ├── shoot.mp3
│           └── music.mp3
├── src/
│   ├── main.tsx              # Entry point
│   ├── style.css             # Canvas and body styles
│   ├── runtime-assets.ts     # Optional loadTexture / loadSound calls
│   ├── actions.ts            # Input.createAction — all input bindings
│   ├── constants.ts          # Game-wide constants (GRAVITY, PLAYER_SPEED, etc.)
│   ├── scenes/
│   │   ├── menu.tsx
│   │   └── game.tsx
│   ├── components/
│   │   ├── player.tsx
│   │   ├── enemy.tsx
│   │   └── hud.tsx
│   └── lib/                  # Utilities and helpers
├── index.html
├── tsconfig.json
└── vite.config.js
```

### Why this layout works

**`src/actions.ts`** — `Input.createAction` throws a `DuplicateKeyError` if the same key combo is registered twice. Defining all actions in a single module and importing the symbols ensures they are created exactly once.

**`src/constants.ts`** — Keeps magic numbers out of components. Values like `GRAVITY`, `PLAYER_SPEED`, or `TILE_SIZE` live in one place and are easy to tweak without hunting through the codebase.

**`src/assets.ts`** — Centralizing static `?texture` and `?sound` imports means the rest of your code imports typed symbol constants (`import { PLAYER_TEX } from '../assets.js'`) rather than hard-coding paths. Use runtime loaders only for URLs that are not known at build time.

**`src/scenes/`** — Each scene is its own file. Scene component functions can be large, and keeping them isolated prevents `main.tsx` from becoming a thousand-line file.

**`src/components/`** — Reusable game entities (player, enemy, HUD elements) live here. A component is just a function that returns JSX, so there is no technical distinction between a "scene" and a "component" beyond convention.

**`src/lib/`** — Utility functions, math helpers, or wrapper code that doesn't belong to a specific component. Keep it flat unless the project grows past ~20 files, then consider subfolders.

**`src/style.css`** — Vite injects this automatically via the import in `main.tsx`. Use it to reset body margins, set canvas background color, or add CSS custom properties.

## Actions file

```ts src/actions.ts
import { Input } from 'fraxel'

export const MoveLeft = Input.createAction({ key: 'ArrowLeft' })
export const MoveRight = Input.createAction({ key: 'ArrowRight' })
export const Jump = Input.createAction({ key: ' ' })
export const Shoot = Input.createAction({ key: 'f' })
```

Import these symbols wherever you need them. Because `Input.createAction` is called at module load time, the actions are always registered before any component runs.

## Constants file

```ts src/constants.ts
export const GRAVITY = 980
export const PLAYER_SPEED = 240
export const JUMP_FORCE = -480
export const TILE_SIZE = 16
```

Keep constants as plain exports. If a value needs to be reactive (changes at runtime), it belongs in a signal inside the component, not in this file.

## Asset file

```ts src/assets.ts
export { default as PLAYER_TEX } from './assets/sprites/player.png?texture'
export { default as ENEMY_TEX } from './assets/sprites/enemy.png?texture'
export { default as SHOOT_SFX } from './assets/sounds/shoot.mp3?sound'
export { default as BG_MUSIC } from './assets/sounds/music.mp3?sound'
```

The Vite plugin resolves imports with `?texture` to a texture ID and imports with `?sound` to a sound ID. Import the symbols wherever your components need them.

For dynamic URLs that are not known at build time, use `loadBatchAsset` to load groups of runtime assets with a progress callback:

```ts src/assets.ts
import { loadBatchAsset } from 'fraxel'

const [PLAYER_TEX, ENEMY_TEX] = await loadBatchAsset('texture', [
  '/assets/sprites/player.png',
  '/assets/sprites/enemy.png',
])

export { PLAYER_TEX, ENEMY_TEX }
```

### Direct asset imports

You can also import assets directly in your components instead of centralizing them in `assets.ts`:

```tsx src/components/player.tsx
import PLAYER from '../assets/sprites/player.png?texture'

export function Player() {
  return <sprite textureId={PLAYER} />
}
```

This eliminates the need for a separate `assets.ts` file, though you can still use both patterns together - query imports for static assets and `loadTexture` / `loadBatchAsset` for runtime loading.

See the [Vite Plugin Reference](/api/engine/vite-plugin#asset-imports) for supported formats and details.

## Optional: per-scene loaders

For large projects with many assets per scene, you can create a `loaders/` directory with one file per scene. Each file re-exports the assets that scene needs:

```ts src/loaders/game-loader.ts
export { default as PLAYER_TEX } from '../assets/sprites/player.png?texture'
export { default as ENEMY_TEX } from '../assets/sprites/enemy.png?texture'
export { default as SHOOT_SFX } from '../assets/sounds/shoot.mp3?sound'
```

```tsx src/scenes/game.tsx
import { PLAYER_TEX, ENEMY_TEX } from '../loaders/game-loader.js'

export function GameScene() {
  return (
    <transform>
      <sprite textureId={PLAYER_TEX} />
    </transform>
  )
}
```

This pattern keeps asset loading scoped to the scene that uses them. The trade-off is more files and explicit imports — for most projects, a single `assets.ts` or direct imports with the plugin are simpler.

## Import path overview

Everything is exported from a single entry point — `'fraxel'`:

```ts
import { createGame, GameRoot, SceneRoot, List, Fragment } from 'fraxel'
import { useSignal, useSprite, useTrigger, useEffect } from 'fraxel'
import { loadTexture, shapes, Vector2, Input } from 'fraxel'
```

## How scenes and components relate

In Fraxel there is no technical difference between a scene and a component. Both are functions that return JSX. The distinction is organizational:

- A **scene** is registered with `<SceneRoot name component>` inside `<GameRoot>`. The engine instantiates it when `useScene().change()` is called. It typically returns a root `<transform>` containing the entire game world.
- A **component** is a function you define and call from inside a scene (or another component). It encapsulates a reusable piece of the node tree — a player, an enemy, a HUD element.

```tsx src/scenes/game.tsx
import { Player } from '../components/player.js'
import { HUD } from '../components/hud.js'
import { BG_TEX } from '../assets.js'

export function GameScene() {
  return (
    <transform>
      <sprite textureId={BG_TEX} />
      <Player />
      <HUD />
    </transform>
  )
}
```

## Entry point

Your `main.tsx` should do three things and nothing else:

1. Import `assets.ts` to trigger asset loading (without plugin).
2. Import scene components.
3. Call `createGame` and `game.play()`.

```tsx src/main.tsx
import './style.css'
import './assets' // Side-effect: loads all textures and sounds (without plugin)
import { MenuScene } from './scenes/menu.js'
import { GameScene } from './scenes/game.js'
import { createGame, GameRoot, SceneRoot } from 'fraxel'

const game = createGame(
  <GameRoot width={320} height={180} defaultScene='menu'>
    <SceneRoot name='menu' component={MenuScene} />
    <SceneRoot name='game' component={GameScene} />
  </GameRoot>,
  document.querySelector('#app')!,
)

game.play()
```

<Note>
  Scenes can also be lazy-loaded by passing an async import as the `component`
  prop: `component={() => import('./scenes/game')}`. Fraxel resolves the default
  export automatically. This keeps the initial bundle small for games with many
  scenes.
</Note>
